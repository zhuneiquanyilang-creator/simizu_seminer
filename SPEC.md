## 1. 目的・背景

物理の参考書を数人で **輪読** している。各メンバーが章を分担して
**レジュメ（レジェメ）** を作成し、発表している。

このサイトでは以下を実現したい:

1. **担当管理** — 誰がどの章・節を担当するかを一覧で確認できる
2. **データベース** — レジュメを補完する情報（補足説明・参考文献・式の導出・リンク等）を蓄積・検索できる

---

## 2. 想定ユーザー

**前提: サイト全体がログイン必須（クローズド）。未ログインでは何も見られない。**

**メンバー数: 7人**（2026-06-25 確定）

| 区分 | 人数 | できること |
|------|------|-----------|
| メンバー（ログイン済み） | 7名 | 担当一覧・DBの閲覧、レジュメ／補足情報の投稿・編集・アップロード |
| 管理者 | 上記のうち1〜2名 | 上記に加え、章立ての設定、データ整理 |

**✅ アカウント方式 = 共有パスワード**（2026-06-25 確定）
全員が同じ合言葉でログインする。個別アカウント登録は不要で一番手軽。

実装方針（Supabase）:
- サイト全体に **パスワードゲート** を1枚かける（合言葉が正しければ閲覧・編集可）。
- もしくは Supabase に共有アカウントを1つ作り全員で使用。

⚠️ **共有パスワード方式の注意点**:
- ログインからは「誰が」編集・投稿したか自動では分からない。
  → 担当者名・投稿者名は **手入力の表示用フィールド** として持つ（[4章](#4-データ構造案)）。
- パスワード流出時は変更が必要。身内限定なので運用でカバーする想定。

> ℹ️ 「管理者」も同じパスワードでログインする。役割は運用上の区別
> （章立て編集やデータ整理を主に担う人）にとどめる。

## 3. 主要機能

### 3.1 担当管理（スケジュール）
- 章・節ごとの担当者を一覧表示
- 発表予定日・発表済み／未発表のステータス表示
- 「次回は誰の担当か」がすぐ分かるトップ表示
- 自分の担当に絞り込み

### 3.2 データベース（レジュメ補完）
- 章・節に紐づく補足情報の登録
  - テキスト（Markdown で数式 `$...$` / `$$...$$` が書けると望ましい）
  - 参考文献・外部リンク
  - レジュメ本体（**PDFのみ**）のアップロード
- キーワード・章・担当者での **検索 / 絞り込み**
- タグ付け（例: 「電磁気」「演習問題」「要復習」）

### 3.3 その他（任意・優先度低め）
- コメント機能（疑問点を投稿し合う）
- 更新履歴・新着表示

### 3.4 リリース計画（MVP / 後回し）

ヒアリングでは MVP として **レジュメアップロード** を選択。
ただし担当一覧は本サイトのもう一つの主目的（誰がどこ担当かの確認）であり、
データ（[`data/toc.md`](data/toc.md)）も既に揃っているため、軽量に同梱できる。

**✅ MVP = 担当一覧 ＋ レジュメアップロードの両方**（2026-06-25 確定）

| 優先度 | 機能 | 備考 |
|--------|------|------|
| **MVP** | 担当一覧・ステータス（3.1） | toc.md のデータを表示。7人で節を分担 |
| **MVP** | レジュメアップロード（3.2） | 節に紐づけてPDFを保存・閲覧 |
| 後回し | 補足メモ・タグ・検索（3.2の一部） | DB蓄積が増えてから |
| 後回し | コメント、更新履歴（3.3） | |
| 後回し | 数式表示(LaTeX) | 確定方針どおり後回し |

## 5. 技術構成（更新）

### 採用方針: 静的フロントエンド + 無料BaaS

「ログイン必須」「ファイルアップロード」を**無料**で実現するため、
純粋な静的サイトではなく、**静的サイト + BaaS無料枠** の構成とする。
運用感は静的サイトに近く（サーバー管理不要）、認証・データ保存・
ファイル保存だけをクラウドサービスに任せる。

```
[ブラウザ]
   │  静的ファイル(HTML/CSS/JS) … GitHub Pages / Cloudflare Pages 等で無料ホスト
   ▼
[BaaS（無料枠）]
   ├─ 認証(Auth)        … ログイン管理
   ├─ データベース(DB)  … 章立て・担当・補足エントリ
   └─ ストレージ        … レジュメPDFのアップロード保存
```

### 採用BaaS: Supabase（確定）

| 機能 | Supabase での実現 | 無料枠の目安 |
|------|------------------|------------|
| 認証 | 共有パスワードによるゲート（[2章](#2-想定ユーザー)） | – |
| DB | PostgreSQL に 章・節・担当・補足エントリを格納 | 500MB |
| ストレージ | レジュメPDFを保存 | 1GB |

ホスティングは GitHub Pages / Cloudflare Pages / Vercel 等の無料枠を想定。

### ⚠️ 無料枠の注意点
- レジュメPDFを溜め続けるとストレージ上限（1GB）に達する可能性。
  → 1ファイルあたりのサイズ上限や、古いファイルの扱いを決めておくと安心。
- 無料枠は一定期間アクセスがないとプロジェクトが休止することがある（再開は可能）。
